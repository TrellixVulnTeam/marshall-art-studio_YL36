Configuration Options

This section outlines the mechanics for altering the various aspects of upload writer functionality from its default out of the box functionality. Many of the aspects can be altered entirely through configuration, which should always be a goal of the application development team. Other aspects may require extension to achieve the desired end state. The following table provides an inventory of some of the aspects of upload writer functionality that are alterable, the default behavior and how to modify that behavior.

4 Capturing the various XML documents managed by APDataCollection to disk Determines the existence of an application property named debug_upload_transformations_dump_path. This property, if present, dictates the directory that all views managed by the outgoing APDataCollection are written to.

Configuration via application property:

debug_upload_transformations_dump_path

If the application property is missing, then this feature is turned off.

Programatically via the protected method:

ImportBaseServlet.

writeAPDataToDebugOutput

debug_upload_transformations_dump_path=

${output_dir}upload_transformations

 
5 Controlling which original document is selected as "system of record" original document

This is controlled by a combination of internal state and TDF/LOB/application property named original_document_system_of_record.

If the upload writer TDF transaction does not support original values via the method Transaction.supportsOriginalValues(), then the application property is ignored and the original document is nulled out.

If the id retention parameter is true and the value of this property is front_door_document, then the original "front door" document is used; otherwise, a clone of the current document before data corrections serves as the system of record original document.

Configuration via TDF/LOB/application property:

original_document_system_of_rocord

Valid values:

  • front_door_document
  • current_document_before_data_corrections (default value)

Programmatically via the protected method:

UploadDataManager.

getOriginalDocumentSystemOfRecord

application.original_document_system_of_record=front_door_document If an application needs to vary this setting by LOB or TDF, then the TDF based application property has precedence over the LOB based application property, which has precedence over the application based application property.

Configuration via TDF/LOB/application property: should_retain_id_attributes

Valid values:

  • false (default value)
