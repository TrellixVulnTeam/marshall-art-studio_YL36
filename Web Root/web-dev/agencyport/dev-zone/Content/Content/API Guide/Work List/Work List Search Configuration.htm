Work List Search Configuration

Users can filter each work list column by a corresponding section in the search accordion. The filter type used in that section (i.e., check boxes) can be configured to work smoothly with the type of data in the work list column. You can configure a search accordion to only include sections for the work list columns you want and the interface contents of those sections. Work list search is configured via the SQL statements used to create the work list itself. The following are the tables required to configure work list search:

work_list table

The search_enabled column of the work_list table determines whether search is turned on or off.

  • A value of 1 enables search.
  • A value of 0 (or no value) indicates that the old work list column filter functionality (pre 4.2) is used.

A given work list may use search or the old column filters, but not both.

work_list_column table

The work_list_column table determines the type of filter used for each work list column. The type column defines the type of filtering for a given work list column. The type column has the following types: numeric, string, date, timestamp and Boolean.

If you do not want to implement the search accordion, and instead want to use the old individual column filters, set the search_enabled column in the work_list table to 0. If you decide to go this route, set the values for the following database columns:

  • filter_enabled - Determines whether a column is filter-enabled. A 0 indicates no, 1 indicates yes.
  • sort_enabled - Determines whether a column is sort-enabled. A 0 indicates no, 1 indicates yes.

To use the work list search feature, you must set the search_enabled column of the work_list table to 1. Work list columns are given an associated search accordion section if the filter_enabled column of the work_list_column take has a value of 1. A value of 0 prevents the work list column from displaying in the work list search accordion.

If the checkboxes_required column value is 0, then the filter type becomes the same as the column type (i.e., numeric, date, string, etc). The search accordion section associated with the work list column displays the user interface unique to that filter type.

EXAMPLE:

A string filter type creates a search accordion section with a select list with choices for "is" and "contains" and a text field.

The following table describes what the user sees in the search accordion section for each filter type:

Filter Type Select list options Text field?
Numeric equals, is greater than, is less than, between* Yes
String contains, is Yes
Date on, before, after, between* Yes
Boolean equals Yes
Checkbox N/A N/A

*When the user selects "between," a second text file displays in the accordion section.

If the checkboxes_required column value is 1, the check box filter type is used. The search accordion section associated with the work list column displays a set of check boxes. Only work list columns of type string may use a check box filter type.

If a column is defined as using a check box, it must also define where the value and UI display values for where the check boxes are located. The code_list_id and code_list_column columns provide that information. The code_list_column column determines which code_list table column is the source of the check box display value. The code_list_id corresponds to a code list defined in the code_list table.

code_list table

The code_list table allows you to store locale-specific lists in a generic table.

This table contains code list representations of line of business codes and transaction types.

work_list_search_detail table

The work_list_search_detail table provides a way to specify the specific entries for a given work list column's search accordion section.

These entries are derived from the code_list table. You can create a sublist that contains search accordion section specific values. The transaction type work list column offers a useful example. The following SQL statements configure a work list column to use a check box filter type. The check box gets its display value column in the code_list table. We set the checkboxes_required column to 1. This indicates you want to display the values as check boxes in the search accordion section. Next, specify the code list to use for the source by setting the code_list_id value. Finally, use code_list_column, specify that we want to value from the code list to display tot he user.

EXAMPLE:

INSERT INTO work_list_column(work_list_column_id, work_list_id, table_alias, column_name, column_alias, type,filter_enabled, column_enabled, sort_enabled, column_number, identity_column, work_list_column_detail_id, checkboxes_required, code_list_id, code_list_column)    
VALUES (110,  100,  'work_item',  'transaction_type', 'Transaction Type', 'string', 1, 1,1,10,0,110, 1,100, 'value');

Incorporating a New Line of Business into Search

There are cases when a new line of business needs added to an existing project. You must perform work to include the new line with the search accordion. The following steps show how this is done.

In this example, use the aviation line of business (LOBCD=AVIA). The work list column LOB is defined as using the check box filter type.

Add AVIA to the code_list table. The code_list_id of 200 is the LOB code list for the sdk.

EXAMPLE:

INSERT INTO code_list(code_list_id, locale_id, value, title)
	VALUES (200, 1, 'AVIA', ‘Aviation Line of Business’);
